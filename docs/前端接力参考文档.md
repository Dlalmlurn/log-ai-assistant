# 前端接力参考文档

本文面向接下来负责前端的两位组员，目标是让大家能在当前 React 工作台基础上继续开发，而不是重新起一套页面。

## 1. 当前前端定位

正式前端是 `React + TypeScript + Vite`，用于承载项目最终验收中的安全运营工作台。

必须遵守的产品边界：

- 前端只通过 FastAPI 读取数据。
- 前端不得直连 Elasticsearch、Kafka、本地日志文件或日志生成器。
- 前端不得生成核心业务数据或伪造验收数据。
- 页面展示应围绕真实链路：`Filebeat -> Kafka -> Flink -> Elasticsearch -> FastAPI -> React`。
- Streamlit 只作为历史 MVP 或调试入口，不作为正式前端继续扩展。

相关文档：

- `docs/01_product_shape.md`：产品页面和用户角色。
- `docs/05_api_contract.md`：前后端接口契约。
- `docs/06_acceptance_checklist.md`：最终验收标准。

## 2. 项目位置与运行方式

前端目录：

```text
frontend/
├── index.html
├── package.json
├── vite.config.ts
├── src/
│   ├── App.tsx
│   ├── api.ts
│   ├── main.tsx
│   ├── styles.css
│   └── types.ts
```

安装依赖：

```bash
cd frontend
npm install
```

启动开发服务：

```bash
npm run dev
```

默认 Vite 地址：

```text
http://127.0.0.1:5173
```

后端代理配置在 `frontend/vite.config.ts`：

```ts
proxy: {
  "/api": {
    target: process.env.VITE_API_PROXY_TARGET ?? "http://127.0.0.1:8000",
    changeOrigin: true
  }
}
```

如果 FastAPI 不在 `8000` 端口，可以这样启动前端：

```bash
VITE_API_PROXY_TARGET=http://127.0.0.1:8001 npm run dev
```

生产构建检查：

```bash
npm run build
```

## 3. 当前已实现页面

当前 `frontend/src/App.tsx` 中已有两个正式页面。

### 3.1 实时日志页

导航名称：`Realtime Logs`

对应需求：

- `REQ-002` 日志结构化处理
- `REQ-006` 可视化交互

调用接口：

```text
GET /api/v1/logs
```

当前能力：

- 展示结构化日志表格。
- 支持筛选 `source_type`、`username`、`src_ip`、`status`、时间范围。
- 支持 `limit` 和 `offset` 分页。
- 支持 10 秒轮询刷新。
- 支持暂停和手动刷新。
- 展示 `risk_tags`、状态颜色、日志事件时间和事件 ID。

注意：

- 页面默认查后端最近 24 小时数据。
- 空数据时不要改成 mock 数据，应提示检查 Filebeat、Flink、Elasticsearch 是否有当前数据。

### 3.2 系统状态页

导航名称：`System Status`

对应需求：

- `REQ-001` 日志采集
- `REQ-002` 日志处理
- `REQ-007` 测试与复现

调用接口：

```text
GET /api/v1/health
```

当前能力：

- 展示 Kafka、Flink、Elasticsearch、DashScope 配置状态。
- 展示最近日志入库时间。
- 展示 Kafka consumer lag。
- 每 15 秒自动刷新。

注意：

- `DashScope` 未配置时可以显示为 unavailable，这不等于系统完全不可用。
- 评审时更关键的是 Kafka、Flink、Elasticsearch 和最新入库时间。

## 4. 当前代码职责

### `frontend/src/App.tsx`

当前主页面和组件都在这里：

- 页面状态：`page`
- 页面组件：`RealtimeLogsPage`、`SystemStatusPage`
- 通用组件：`PageHeader`、`ServiceCard`、`Metric`、`StatusPill`、`ErrorBanner`、`EmptyState`
- 工具函数：日期格式化、错误格式化、状态颜色判断

后续页面多了以后，建议逐步拆分：

```text
src/
├── pages/
│   ├── RealtimeLogsPage.tsx
│   ├── SystemStatusPage.tsx
│   ├── AlertsPage.tsx
│   ├── BaselinesPage.tsx
│   ├── AiReportsPage.tsx
│   └── DailyReportsPage.tsx
├── components/
│   ├── PageHeader.tsx
│   ├── StatusPill.tsx
│   ├── ErrorBanner.tsx
│   └── EmptyState.tsx
├── api.ts
├── types.ts
└── styles.css
```

拆分不是当前第一优先级。优先把功能闭环做出来，文件过大时再拆。

### `frontend/src/api.ts`

所有接口请求必须放在这里或由这里分出去。

已有方法：

```ts
fetchHealth()
fetchLogs(query)
```

已有错误类型：

```ts
ApiRequestError
```

新增页面时，不要在组件里直接写散乱的 `fetch("/api/...")`。建议新增类似方法：

```ts
export async function fetchAlerts(query: AlertsQuery, signal?: AbortSignal) {
  return apiFetch<ListResponse<AlertEvent>>(`/api/v1/alerts?${params.toString()}`, { signal });
}
```

### `frontend/src/types.ts`

前端类型集中放这里，必须尽量和 `docs/05_api_contract.md`、后端 Pydantic model 对齐。

已有类型：

- `SourceType`
- `HealthResponse`
- `NormalizedLog`
- `ListResponse<T>`
- `LogsQuery`
- `ApiError`

新增接口前，先加类型，再写 API 方法，再写页面。

### `frontend/src/styles.css`

当前样式是偏安全运营工作台风格：

- 左侧导航
- 主内容工作区
- 表格为主
- 卡片只用于状态项和指标项
- 色彩克制，优先可读性和信息密度

后续页面请保持一致，不要改成营销页、海报页或大面积装饰风格。

## 5. 待开发页面建议

以下页面来自 `docs/01_product_shape.md` 和 `docs/05_api_contract.md`。

### 5.1 异常事件页

优先级：最高

对应接口：

```text
GET /api/v1/alerts
GET /api/v1/alerts/{alert_id}
POST /api/v1/alerts/{alert_id}/analyze
```

页面目标：

- 展示告警列表。
- 支持按风险等级、用户、规则、状态、时间过滤。
- 点击告警后展示详情。
- 详情里展示 `rule_hits`、`evidence`、`related_event_ids`、`baseline_deviations`。
- 支持触发 AI 研判。

建议布局：

- 上方筛选栏。
- 左侧或上方告警表格。
- 右侧或下方详情面板。
- 高风险告警用清晰标签标出，但不要只靠颜色表达含义。

### 5.2 用户基线页

优先级：高

对应接口：

```text
GET /api/v1/baselines
GET /api/v1/baselines/{username}
POST /api/v1/baselines/rebuild
```

页面目标：

- 展示用户 baseline 列表。
- 展示单个用户画像：活跃时间、常用 IP、常用资源、失败登录数、敏感访问率。
- 展示该用户当前偏离项。
- 支持触发 baseline 重建。

建议先做列表和详情，不必一开始做复杂图表。

### 5.3 AI 研判页

优先级：中高

对应接口：

```text
GET /api/v1/ai-reports
POST /api/v1/alerts/{alert_id}/analyze
```

页面目标：

- 展示 AI 报告列表。
- 展示攻击类型、风险等级、原因、建议、置信度、后续动作。
- 明确显示是否为 mock fallback。

注意：

- 无 API Key 时，mock 输出必须明确标识，不能包装成真实模型结果。

### 5.4 每日简报页

优先级：中

对应接口：

```text
GET /api/v1/daily-reports
POST /api/v1/daily-reports
```

页面目标：

- 展示日报列表。
- 展示某日安全评分、高危用户、主要风险、建议动作。
- 支持指定日期生成日报。

### 5.5 概览页

优先级：可后置

对应接口：

```text
GET /api/v1/stats/overview
GET /api/v1/stats/rules
GET /api/v1/stats/users/risk
```

页面目标：

- 展示今日日志量、告警数、高危告警数、活跃用户数。
- 展示规则命中排行。
- 展示高风险用户排行。

如果时间不够，概览页可以放到异常事件和 baseline 之后。

## 7. 新增一个页面的推荐流程

以“异常事件页”为例：

1. 在 `src/types.ts` 增加接口类型。

```ts
export type RiskLevel = "低" | "中" | "高" | "紧急";

export type AlertEvent = {
  alert_id: string;
  event_time: string;
  username?: string | null;
  risk_level: RiskLevel;
  risk_score: number;
  status: "new" | "analyzed" | "closed";
  rule_hits: string[];
  evidence: Record<string, unknown>;
  related_event_ids: string[];
};
```

2. 在 `src/api.ts` 增加请求方法。

```ts
export async function fetchAlerts(query: AlertsQuery, signal?: AbortSignal) {
  const params = new URLSearchParams();

  Object.entries(query).forEach(([key, value]) => {
    if (value !== undefined && value !== null && value !== "") {
      params.set(key, String(value));
    }
  });

  return apiFetch<ListResponse<AlertEvent>>(`/api/v1/alerts?${params.toString()}`, { signal });
}
```

3. 在 `App.tsx` 或 `src/pages/AlertsPage.tsx` 增加页面组件。

需要包含：

- 初始 query。
- `loading`、`error`、`updatedAt`。
- 筛选表单。
- 表格或列表。
- 空状态。
- 分页。

4. 在侧边栏增加导航项。

注意 `PageKey` 类型也要同步增加：

```ts
type PageKey = "logs" | "status" | "alerts";
```

5. 对照 `docs/05_api_contract.md` 检查字段和接口是否一致。

6. 跑构建：

```bash
npm run build
```

## 8. 前端验收自查清单

每次提交前请确认：

- [ ] 页面只调用 FastAPI。
- [ ] 没有新增核心业务 mock 数据。
- [ ] 接口错误时有可读提示。
- [ ] 空数据时有空态提示。
- [ ] loading 状态不会造成页面跳动或空白。
- [ ] 表格字段和接口类型一致。
- [ ] 页面能在常见桌面宽度正常显示。
- [ ] `npm run build` 通过。
- [ ] 新页面能追溯到至少一个 `REQ-*`。

## 9. 汇报时可以强调的前端进展

当前前端已经从“临时展示”转向“正式工作台”：

- 采用 React + TypeScript + Vite。
- 通过 FastAPI 统一访问后端。
- 已有实时日志页，能对接 ES 中结构化日志。
- 已有系统状态页，能展示 Kafka、Flink、ES、DashScope 和 consumer lag。
- 页面结构已经为异常事件、用户基线、AI 研判、日报继续扩展预留空间。

下一阶段前端重点不是美化，而是把安全分析闭环补齐：

```text
实时日志 -> 异常事件 -> baseline 解释 -> AI 研判 -> 日报沉淀
```
