# 数据规模与场景规格

本文件定义项目所需的数据规模、日志生成原则、用户画像和安全场景。

数据生成不是为了堆数量，而是为了让行为 baseline 和异常评分具备可信基础。

## 数据规模要求

项目每日原始增量日志不少于 1GB。

这里的 1GB 指进入采集链路前的原始 JSON 或日志文本大小，不是 ClickHouse 压缩后的大小。

系统必须能记录：

- 当日原始日志大小。
- 当日原始日志条数。
- Kafka `raw_logs` 接收条数。
- Flink 成功解析条数。
- ClickHouse 入库条数。
- 字段缺失率。
- 异常样本数量和高危异常样本数量。
- ClickHouse 表占用空间。
- 实测压缩率。
- 核心查询耗时。

## 生成器来源边界

导师提供的 VPN log-generator 可以作为第一类数据源，用于提供 VPN 登录日志、用户画像先验和基础异常样本。

但项目目标数据集不局限于 VPN 日志。VPN 日志不能代表最终数据集的全部日志类型，也不能替代 OA、API、系统、文件、数据库和安全设备日志。

如果某类日志不是原始生成器直接提供，应通过扩展生成脚本或适配器生成同一数据契约下的标准日志。

## 持续增量生成要求

项目允许在不修改原始生成器核心逻辑的前提下增加外层调度器。

外层调度器的作用是持续生成增量日志，并将日志追加到 Filebeat 可采集的目标文件中。

该机制只负责持续生成和文件追加，不代表已经满足数据规模、字段质量、多日志类型或攻击过程要求。

正式采集入口仍然是 Filebeat。Python Producer 只作为辅助工具。

## 日志生成原则

日志生成器必须模拟企业环境中的长期行为，而不是随机生成互不相关的样例。

必须具备：

- 多用户。
- 多角色。
- 多主机。
- 多来源 IP。
- 多日志类型。
- 工作日和非工作日差异。
- 白天和夜间差异。
- 正常行为周期。
- 少量异常注入。
- 可追踪的攻击过程。

## 用户画像

至少包含以下账号类型：

| 类型 | 正常行为 |
| --- | --- |
| 普通员工 | 工作时间登录，访问 OA、API、文件系统。 |
| 财务人员 | 访问财务系统，下载报表，较少夜间行为。 |
| 管理员 | 访问管理接口，执行配置变更，来源较稳定。 |
| 运维人员 | 访问主机和系统日志，可能在维护窗口操作。 |
| 服务账号 | 自动化调用，时间规律强，来源主机固定。 |
| 离职风险用户 | 下载量上升，访问平时较少访问的资源。 |
| 攻击者账号 | 异常来源、异常时间、异常行为组合。 |

生成器中的用户画像是数据生成先验。系统 baseline 必须从正式入库数据中统计得出。

## 日志类型

至少覆盖：

| 类型 | 示例行为 |
| --- | --- |
| VPN 日志 | 登录、登出、失败登录。 |
| OA 日志 | 页面访问、审批、文件访问。 |
| API 日志 | 接口调用、导出、错误请求。 |
| 系统日志 | SSH、进程启动、服务异常。 |
| 文件日志 | 文件读取、下载、删除。 |
| 数据库日志 | 查询、导出、权限变更。 |
| 安全设备日志 | 阻断、告警、异常连接。 |

## VPN 日志 5W1H 映射

VPN 日志可作为第一类 5W1H 映射样例。

| 五W1H | 字段示例 | 说明 |
| --- | --- | --- |
| Who | 用户名、部门、角色、账号类型 | 谁在登录。 |
| When | 日志时间 | 什么时候登录。 |
| Where | 来源 IP、来源国家城市、VPN 网关、目标内网地址 | 从哪里登录到哪里。 |
| What | 登录成功、登录失败、登出、会话事件 | 做了什么。 |
| Why | 失败原因、风险标签、维护窗口、白名单 | 为什么可能异常或为什么可以解释。 |
| How | 协议、认证方式、客户端、会话 ID | 如何完成登录。 |

## 正常场景

至少覆盖：

- 普通员工在工作时间从常用 IP 登录并访问 OA。
- 财务人员在工作时间访问报表系统。
- 运维人员在维护窗口访问服务器。
- 服务账号按固定周期调用 API。
- 管理员从固定来源执行少量配置变更。

## 异常场景

至少覆盖：

| 场景 | 典型行为 | 期望风险 |
| --- | --- | --- |
| 暴力破解 | 同一账号短时间多次失败登录。 | medium 到 high。 |
| 凭证填充 | 同一 IP 尝试多个账号。 | high。 |
| 账号接管 | 新 IP 登录成功后访问敏感资源。 | high 到 critical。 |
| 数据窃取 | 短时间大量下载或导出敏感数据。 | high 到 critical。 |
| 权限滥用 | 普通账号访问 admin、config、backup。 | high。 |
| 服务账号异常 | 服务账号从异常主机或异常时间登录。 | medium 到 high。 |
| 横向移动 | 同一来源短时间访问多个内网目标。 | high。 |

## VPN 大流量的语义边界

VPN 会话中的大流量可以作为数据外传风险信号。

但它不能替代独立的文件下载、数据库导出或 API 导出日志。数据窃取场景应至少包含一种独立资源访问、下载或导出类日志。

## 异常比例

建议比例：

| 类型 | 比例 |
| --- | --- |
| 正常日志 | 95% 到 99% |
| 普通异常 | 1% 到 3% |
| 高危异常 | 0.1% 到 0.5% |

比例可配置，但必须记录实际注入数量。

## 攻击过程可追踪

用于模型训练、质量评估和异常分析的数据，应支持可追踪场景标识。

同一攻击过程中的多条日志应共享 `attack_chain_id`，并通过 `step_index` 标明顺序。

建议字段：

| 字段 | 说明 |
| --- | --- |
| `scenario_id` | 场景 ID。 |
| `attack_chain_id` | 攻击过程 ID。 |
| `step_index` | 攻击过程中的步骤序号。 |
| `scenario_type` | 场景类型。 |
| `injected_label` | 生成样本标签。 |

每个高危异常场景应能追踪：

```text
原始日志
  -> 标准化日志
  -> 规则命中或 baseline 偏离
  -> 异常事件
  -> AI 研判
  -> 工作台查询结果
```

这里强调的是数据可追踪性，不是一次性页面效果。

## 数据质量要求

字段质量至少记录：

- `event_time` 缺失率。
- `user_id` 缺失率。
- `src_ip` 缺失率。
- `action` 缺失率。
- `result` 缺失率。
- `raw_log` 缺失率。
- 无法解析日志比例。

关键字段缺失率过高时，不能直接用该数据训练 baseline。

## 数据对账要求

必须能对账：

```text
generated_count
injected_anomaly_count
injected_high_risk_count
raw_logs_count
parsed_logs_count
clickhouse_insert_count
security_logs_count
```

允许因为解析失败或过滤产生少量差异，但差异必须能解释。

`clickhouse_insert_count` 表示写入 ClickHouse sink 的记录数，`security_logs_count` 表示最终进入主日志事实表 `security_logs` 的记录数。若两者不同，必须说明原因。

## 压缩率记录

需要记录原始大小和 ClickHouse 表大小。

压缩率作为实测结果呈现，不作为未经验证的固定指标。
