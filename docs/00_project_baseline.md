# 项目开发基准

本文件定义项目的最高基准。后续功能、页面、接口、数据表、规则和模型设计都必须能追溯到本文件的需求编号。

如实现方案和本文冲突，必须先修改本文或追加 ADR，再进入主线开发。

## 项目定位

本项目是一个面向企业安全运营场景的日志分析 AI 助手原型。

它的目标不是把日志生成器、查询脚本和大模型总结拼在一起，而是形成一个可持续扩展的安全日志分析产品雏形。系统应能采集结构化安全日志，建立用户行为基线，发现偏离行为，生成可解释的异常事件，并让 AI 在有证据的基础上辅助研判和反馈优化。

核心价值包括：

- 建立用户行为 baseline。
- 发现偏离正常行为的异常。
- 把统计异常转化为可理解、可追踪、可处置的安全事件。
- 用 AI 辅助研判和反馈校准，而不是让 AI 替代检测链路。
- 用足够规模的数据支撑行为建模，而不是只处理少量样例日志。

## 文档定位

本套文档规定项目应该成为什么，而不是规定每一步如何实现。

因此，正式文档只记录目标形态、开发基准、数据契约、架构边界和最终质量标准。不在本文档集中写入某个分支的当前实现差距、临时启动方式或阶段性合并方案。

阶段性实现细节可以进入 issue、开发记录或代码注释，但不能反向改变项目目标形态。

## 当前主线

```text
Filebeat -> Kafka -> Flink -> ClickHouse -> FastAPI -> React
```

ClickHouse 是唯一主存储和分析引擎。

Elasticsearch 不作为运行时依赖。主线代码不继续新增 Elasticsearch sink、index、query adapter 或以 Elasticsearch 为核心的数据契约。

## 需求基线

| ID | 原始要求 | 项目能力 | 目标状态 |
| --- | --- | --- | --- |
| REQ-001 | 日志采集 | 使用 Filebeat 或同等级采集组件完成日志全量和增量采集。 | 新增日志持续进入 Kafka，不以手动导入作为正式路径。 |
| REQ-002 | 日志结构化处理 | 对 VPN、OA、API、系统、文件、数据库、安全设备等日志做标准化处理。 | 原始日志转换为标准 JSON，并进入 ClickHouse 主日志表。 |
| REQ-003 | 行为建模 | 基于历史数据构建用户行为 baseline。 | 每个主要用户有可查询、可更新、可解释的五W1H baseline。 |
| REQ-004 | 智能研判 | 将异常事件和上下文证据发送给大模型，判断攻击类型和风险等级。 | AI 输入包含异常、baseline、相关日志和窗口统计，输出结构化结论。 |
| REQ-005 | 自动化总结 | 生成每日安全态势简报。 | 简报基于真实统计结果生成，并能在前端查询。 |
| REQ-006 | 可视化交互 | 提供安全运营工作台。 | React 通过 FastAPI 展示日志、异常、用户画像、AI 研判和安全态势。 |
| REQ-007 | 场景复现 | 构造正常和攻击场景，支持全流程追踪。 | 至少覆盖 3 类正常场景和 3 类高危攻击场景，并能追踪到异常事件。 |
| REQ-008 | 实时告警与扩展 | 支持准实时异常发现和规则扩展。 | 高危异常能进入异常事件列表，规则具备可配置和可扩展边界。 |

## 技术基线

| 层级 | 当前口径 |
| --- | --- |
| 日志采集 | Filebeat 是正式采集入口。Python Producer 只可作为调试工具。 |
| 消息队列 | Kafka 作为流式传输和缓冲层。 |
| 流处理 | Flink 负责清洗、字段标准化、轻量规则、窗口统计和写入前处理。 |
| 主存储 | ClickHouse 存储结构化日志、用户特征、baseline、异常事件、AI 研判、AI 反馈和日报数据。 |
| 后端 API | FastAPI 对外提供查询、分析、状态、AI 研判和反馈接口。 |
| 前端展示 | React + TypeScript + Vite 构建安全运营工作台。 |
| 行为建模 | 优先基于历史数据 T+1 建模，不在实时流中硬编码 baseline。 |
| 异常输出 | 规则检测、窗口统计和 baseline 偏离必须统一产出 `AnomalyEvent`。 |
| 风险等级 | 内部统一使用 `low`, `medium`, `high`, `critical`，前端可映射为中文展示。 |
| AI 研判 | LLM 只处理高可疑事件的结构化证据包，不分析全量日志。 |

## 主线约束

以下约束进入项目长期主线：

- Filebeat 是正式采集入口。Python Producer 仅用于本地调试、链路隔离验证或测试数据输入。
- Flink 承担日志清洗、字段标准化、轻量规则和窗口统计，不承担完整 T+1 baseline 建模。
- 规则检测、窗口统计、baseline 偏离和评分结果必须统一进入 `AnomalyEvent`，不能分别向前端、AI 或日报输出不同形态的告警对象。
- 新来源 IP、新设备、新来源地等判断必须基于历史 baseline、日级特征或持久化 seen 表，不得只依赖检测进程内存。
- VPN 日志生成器可以作为第一类数据源，但不能代表最终数据集的全部日志类型。
- 生成器中的用户画像是数据生成先验，不等于系统行为 baseline。系统 baseline 必须从入库后的历史日志中统计得出。
- AI 反馈进入反馈表和调优候选流程，不直接自动修改生产规则或 baseline。

## 主线准入规则

新增功能进入主线前必须满足：

- 能追溯到至少一个 `REQ-*` 编号。
- 不绕过正式数据链路生成核心业务数据。
- 不让前端直接读取 Kafka、ClickHouse 或本地日志文件。
- 不把 AI 输出作为无证据的安全结论。
- 不新增 Elasticsearch 运行时依赖。
- 对 baseline、score、rule 的新增逻辑必须说明数据来源和解释方式。

## 非目标

以下内容不作为当前主线目标：

- 用大模型直接读取全量日志并给出安全结论。
- 只靠固定 if-else 规则替代行为建模。
- 只用少量样例日志证明行为分析有效。
- 把前端 mock 数据作为核心业务结果。
- 同时维护 Elasticsearch 和 ClickHouse 两套主存储。
- 把某个分支的当前实现状态写成项目目标。
