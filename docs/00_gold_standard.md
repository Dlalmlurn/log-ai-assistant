# Gold Standard Requirements

本文件把甲方原始需求固化为后续开发的最高依据。来源包括 `docs/初始需求文档.xlsx` 和 `docs/日志分析.xlsx`；如后续实现、演示或技术选择与本文冲突，以本文为准，并通过 ADR 记录变更理由。

## 项目定位

构建一个日志分析 AI 助手，帮助安全人员快速分析企业日志并发现异常行为。项目不是脚本集合，也不是单页演示，而是一个面向安全运营场景的实时日志安全分析产品原型。

核心价值不在于堆叠技术栈，而在于：

- 建立用户行为基线。
- 发现偏离正常行为的异常。
- 将统计异常转化为可理解、可追踪、可处置的安全报告。
- 用 AI 辅助研判，而不是让 AI 替代检测链路。

## Requirement Baseline

| ID | 原始要求 | 必须交付的能力 | 验收信号 |
| --- | --- | --- | --- |
| REQ-001 | 日志采集 | 通过 Filebeat 或同等级采集组件完成日志全量与增量采集。 | 新增日志能持续进入 Kafka，不以手动导入作为正式路径。 |
| REQ-002 | 日志结构化处理 | 对 VPN、OA、API、系统等日志做标准化处理，字段至少包含日志生成时间、用户名、行为、事件、采集时间、入库时间。 | 原始日志能转换为标准 JSON，且写入持久化存储。 |
| REQ-003 | 简单行为建模 | 统计每个用户过去一段时间内的正常习惯，例如活跃时间、API 调用频率、常用 IP 段。 | 每个用户有可查询、可更新、可用于异常解释的 baseline。 |
| REQ-004 | 智能判断 | 将检测到的异常片段和上下文发送给大模型，判断攻击类型并给出风险等级。 | AI 输入包含异常、baseline 和相关日志；输出结构化攻击类型、风险等级、原因、建议。 |
| REQ-005 | 自动化总结 | 每日自动生成《安全态势简报》，总结高危用户及整体安全评分。 | 每日简报可生成、可存储、可在前端查看。 |
| REQ-006 | 可视化交互 | 展示实时日志流、UEBA 异常用户排行、AI 生成的安全处置建议。 | React 工作台能通过后端 API 展示真实入库数据。 |
| REQ-007 | 测试与复现 | 构造正常和攻击场景，完成全流程联调测试与项目文档整理。 | 至少覆盖 3 类正常场景和 3 类高危攻击场景，并能从原始日志追踪到前端展示。 |
| REQ-008 | 实时告警与扩展 | 支持实时异常发现和自定义规则扩展。 | 高危异常能秒级或准实时进入告警列表，规则具备可配置或可扩展边界。 |

## Technology Baseline

| 层级 | 口径 |
| --- | --- |
| 日志采集 | Filebeat 是正式采集入口；Python Producer 只保留为调试兜底。 |
| 消息队列 | Kafka 作为流式传输和缓冲层。 |
| 流处理 | Flink 负责实时清洗、结构化和窗口统计。 |
| 存储检索 | Elasticsearch 作为日志、告警、AI 报告、日报和 baseline 的查询中心。 |
| 后端 API | FastAPI 对外提供查询、分析、日报和健康检查接口。 |
| 前端展示 | React + TypeScript + Vite 构建安全运营工作台。 |
| AI 研判 | LangChain / DashScope / 兼容 LLM 接口用于研判和解释，不负责原始检测。 |

## Traceability Rule

任一新功能、页面、接口、规则或演示脚本必须标注至少一个 `REQ-*` 编号。不能追溯到本文件的功能默认不进入主线开发，除非先更新本文件或新增 ADR。
