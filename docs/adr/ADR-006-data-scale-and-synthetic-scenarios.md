# ADR-006: 每日不少于 1GB 原始日志和企业场景生成

**Status:** accepted

## Decision

项目需要通过脚本生成每日不少于 1GB 的原始增量日志，并模拟多用户、多角色、多日志类型和可追踪异常场景。

导师提供的 VPN log-generator 可以作为第一类数据源，但不能代表最终数据集的全部日志类型。

项目允许增加外层调度器持续生成增量日志并追加到 Filebeat 可采集文件中。该调度器不改变数据契约，也不替代多类型日志和 1GB/day 数据规模要求。

## Context

当前日志量过小，无法支撑真实场景下的行为分析。少量日志会让 baseline、分位数、常用 IP、活跃时间和异常评分缺乏可信度。

项目需要用足够规模的数据证明结构化日志分析和行为建模的有效性。

## Alternatives

| 方案 | 结论 | 原因 |
| --- | --- | --- |
| 继续使用少量样例日志 | 不采用 | 难以支撑行为分析。 |
| 只生成随机日志 | 不采用 | 缺少用户画像和行为连续性。 |
| 只使用 VPN 单类型日志 | 不采用 | 不能覆盖完整安全分析场景。 |
| 生成企业场景日志并注入异常 | 采用 | 能支撑 baseline 和攻击过程追踪。 |

## Rationale

每日不少于 1GB 原始日志可以让系统具备基本数据规模。多角色和多场景数据可以让 baseline 更接近真实企业环境。

VPN 日志具备很好的 5W1H 映射基础，但数据窃取、权限滥用、系统异常和横向移动仍需要文件、数据库、API、系统和安全设备日志支撑。

## Consequences

- 日志生成器必须支持用户画像。
- 日志生成器必须支持正常行为周期。
- 日志生成器必须支持异常注入。
- 目标数据集必须覆盖多种日志类型。
- 高危场景应支持 `scenario_id`、`attack_chain_id` 和 `step_index`。
- 系统必须记录生成量、处理量、入库量和字段质量。
- 压缩率和查询性能必须以实测数据记录。
